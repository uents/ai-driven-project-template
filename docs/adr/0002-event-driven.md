# ADR-0002: イベント駆動アーキテクチャの採用

## ステータス

承認済

## コンテキスト

マイクロサービス間の連携方式を決定する必要がある。サービス間の結合度を低く保ちつつ、信頼性の高い連携を実現したい。

## 決定

Amazon EventBridge を使用したイベント駆動アーキテクチャを採用する。サービス間の同期的なAPI呼び出しは原則禁止とする。

## 理由

- サービス間の疎結合を実現できる
- イベントの再処理・リプレイが可能
- EventBridgeのスキーマレジストリによりイベント契約を管理できる
- 非同期処理により個々のサービスの可用性が独立する

## トレードオフ

- **メリット**: 疎結合、拡張性、耐障害性
- **デメリット**: 結果整合性の許容が必要、デバッグの複雑化、イベント順序の保証なし

## 代替案

| 選択肢 | 評価 |
|---|---|
| 同期API呼び出し | シンプルだがサービス間の結合度が高くなる |
| SQS | キューイングには適するがイベントルーティングの柔軟性が低い |
| SNS | Pub/Subモデルだがフィルタリングの柔軟性でEventBridgeが優位 |
